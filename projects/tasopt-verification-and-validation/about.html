Verification and validation of TASOPT against public aircraft data

Aditeya Shukla · MIT Laboratory for Aviation and the Environment

About Updates

Abstract

This project develops an AI-agent-driven, auditable pipeline for designing, sizing, and validating TASOPT aircraft models against real aircraft. The first case is a Boeing 737-800-like model with CFM56-7B engines. Public airport planning documents and emissions databank records are used to define geometry, mission requirements, fuel capacity, payload-range targets, and LTO thrust/fuel-flow validation points.

The central claim is deliberately narrow: an automated agent workflow should be able to assemble traceable inputs, generate a TASOPT model, run sizing and off-design missions, and check whether the model reproduces aircraft-level mission capability and selected engine operating points within a quantified tolerance. A perfect representation is not the objective; traceability and physically meaningful agreement are.

Scientific questions

Q1

Can public data define a TASOPT aircraft without hidden fitting?

Every input is assigned a provenance category before analysis: observed/certified, derived, assumption, calibration knob, or validation target.

Q2

Does the sized aircraft recover held-out aircraft-level quantities?

MTOW is held out as a validation target. Off-design payload-range points are also used to assess mission capability beyond the sizing point.

Q3

Does the engine model reproduce independent LTO operating points?

EDB thrust and fuel-flow at the four standard LTO points are validation metrics. Emission indices are reference data for now, not predicted targets.

Geometry constrains the model; payload-range tests the mission.

The project links two kinds of public evidence. Airport planning document dimensions constrain the airframe representation. Payload-range diagrams then test whether the sized model captures aircraft-level capability under realistic mission constraints.

Agent system

The research object is the pipeline, not just one aircraft.

Data agent

Extracts APD/EDB values, units, variants, source pages, and digitized payload-range points.

Provenance agent

Classifies each input as observed, derived, assumption, calibration knob, or validation target.

Model agent

Builds the TASOPT TOML and records any documented post-load mutations.

Execution agent

Runs sizing and off-design missions, captures convergence behavior, and exports structured outputs.

Audit agent

Compares against held-out targets and separates model error from digitization uncertainty.

Sizing mission

Selected-payload maximum-range point

The B737-800 model is sized at the APD highest-gross-weight payload-range line point corresponding to maximum range for the selected sizing payload. This point is distinct from both the max-payload range corner and the zero-payload ferry range. It is the payload/range condition where the aircraft is fuel/MTOW-limited for the chosen payload.

Max payload

The left side of the curve is the structural payload limit. Range is small, but the aircraft carries its maximum allowable payload.

Uncertainty and tests

Credibility depends on bookkeeping

Digitization uncertainty

The APD payload-range chart must be digitized with a target uncertainty below 0.5% if possible, and repeated extraction should be used to estimate scatter.

Unit integrity

TASOPT stores many weights as force. Tests will explicitly guard against kg/lbf/N bookkeeping errors.

Mission monotonicity

At fixed range, feasible payload should not increase with stricter constraints; at fixed payload, feasible range should not increase when payload increases.

Write boundaries

Scripts should write outputs under the project workspace and should not edit TASOPT source without permission.

Study plan

Stages of the research

  1. Stage 0TASOPT analysis and input classification

    Build the data ledger and classify every TOML field.

  2. Stage 1AB737-800 reconstruction

    Reconstruct a B737-800-like model with CFM56-7B engines from APD/EDB data and documented assumptions.

  3. Stage 1BSupervised calibration

    Adjust only approved knobs and record every change with provenance and motivation.

  4. Stage 1CB737 V&V

    Validate against MTOW, held-out payload-range points, EDB LTO thrust/fuel-flow, and field-length data if available.

  5. Stage 1DOptional design optimization

    Report fuel-burn optimization separately from validation, relative to the validated baseline.

  6. Stage 2Automated framework

    Generalize the workflow to other aircraft.

  7. Stage 3B777-300ER / GE90

    Apply the automated V&V pipeline to a wide-body aircraft case.
